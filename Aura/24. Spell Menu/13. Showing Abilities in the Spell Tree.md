这份字幕主要讲述了在 Gameplay Ability System (GAS) 中，如何通过 **Gameplay Tags** 设计和实现技能的**状态（Status）**与**类型（Type）**系统，以便在技能菜单（Spell Menu）中正确显示和管理技能。

以下是该段视频的详细标题大纲式总结：

### 一、 技能状态（Ability Status）的设计

为了在技能菜单中正确显示技能（如是否可用、图标颜色等），定义了四种核心状态：

- **1. 已锁定 (Locked)**：
  - 默认状态，不显示技能具体图标（显示锁图标）。
  - 玩家等级未达到要求。
- **2. 可解锁/符合资格 (Eligible)**：
  - 玩家达到了技能所需的等级要求。
  - 显示技能图标，但为**灰色**（Grayed out）。
  - 此时可以消耗技能点（Spell Points）来解锁。
- **3. 已解锁 (Unlocked)**：
  - 已消耗技能点解锁。
  - 显示**彩色**技能图标。
  - 技能可以被装备，但尚未装备到按键上。
- **4. 已装备 (Equipped)**：
  - 技能已分配到具体的输入按键（Input Slot）。
  - 可以在游戏中使用（若蓝量充足）。
  - 在菜单底部的“已装备行”中显示。

### 二、 技能类型 (Ability Type) 的分类

为了区分哪些技能应该显示在技能菜单中，定义了三种类型：

- **1. 攻击型 (Offensive)**：属于攻击系法术，显示在菜单中。
- **2. 被动型 (Passive)**：属于被动系法术，显示在菜单中。
- **3. 无 (None)**：
  - 用于通用能力（如受击反应 Hit React、死亡 Death、事件监听等）。
  - 这些能力始终处于激活状态，**不需要**在技能菜单中显示。

### 三、 C++ 实现：注册 Gameplay Tags

在 `AuraGameplayTags` 单例类中添加新的标签以支持上述逻辑：

- **技能状态标签**：
  - `Abilities.Status.Locked`
  - `Abilities.Status.Eligible`
  - `Abilities.Status.Unlocked`
  - `Abilities.Status.Equipped`
- **技能类型标签**：
  - `Abilities.Type.Offensive`
  - `Abilities.Type.Passive`
  - `Abilities.Type.None`
- **优化项**：顺便添加了 `Abilities.HitReact` 标签，以替换旧的 `Effects.HitReact`，用于受击反应技能。

### 四、 C++ 实现：初始技能状态设置

- **目标**：确保角色出生时获得的初始技能（Startup Abilities）自动处于“已装备”状态。
- **修改位置**：`AuraAbilitySystemComponent` 中的 `AddCharacterAbilities` 函数。
- **逻辑**：在授予技能（`GiveAbility`）之前，向 `AbilitySpec` 的 **Dynamic Ability Tags** 容器中添加 `Abilities.Status.Equipped` 标签。

### 五、 C++ 实现：获取状态的辅助函数

- **创建函数**：在 `AuraAbilitySystemComponent` 中创建静态辅助函数 `GetStatusFromSpec`。
- **功能**：传入一个 `FGameplayAbilitySpec`，返回其当前的技能状态标签。
- **逻辑**：
  - 遍历该 Spec 的 `DynamicAbilityTags`。
  - 使用 `MatchesTag` 检查是否存在匹配 `Abilities.Status` 的标签。
  - 返回找到的状态标签（确保同一时间只有一个状态）。
