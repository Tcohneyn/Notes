### 2. 本地任务的局限性

### 3. GAS 的内置机制

- GAS（Gameplay Ability System）提供了向服务器发送目标数据的机制。
- 当客户端激活 Ability Task 时，服务器稍后也会激活对应任务。
- 定义时间概念：
  - **Epsilon**：服务器 Ability Task 被激活所需时间。
  - **Delta**：RPC 到达服务器所需时间。
- 存在顺序不确定性：
  - 服务器先激活（Epsilon < Delta） → 数据无效。
  - RPC 先到达（Delta < Epsilon） → 数据有效。
- 因为顺序无法保证，所以直接用 RPC 发送可能不可靠。

### 4. GAS 目标数据系统

- 基于 **FGameplayAbilityTargetData** 结构体。
- 内置多种类型的目标数据，可发送到服务器。
- 核心函数：
  - `ServerSetReplicatedTargetData`：将数据发送到服务器。
  - 服务器接收后会广播 **TargetSet Delegate**，触发回调。
- 系统维护 **Ability Target Data Map**，记录 Ability Spec 与目标数据的映射。
- 预测相关：使用 **Prediction Key** 进行客户端预测（暂未详细讲解）。

### 5. 处理同步问题

- 情况 1：Activate 先于 RPC → 数据最终会通过 TargetSet Delegate 获取。
- 情况 2：RPC 先于 Activate → 可能错过广播。
  - 解决方案：调用 `CallReplicatedTargetDataDelegateIfSet` 强制广播，保证服务器获取数据。
